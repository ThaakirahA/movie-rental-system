# 📝 Reflection – Object State and Activity Modeling  
## Aura Reels Movie Rental System

This assignment focused on modelling the dynamic behaviour of the system using state transition diagrams and activity diagrams. While the process was structured, there were several challenges experienced during the design and alignment of these models.

---

## Challenges in Choosing Granularity

One of the main challenges was deciding the level of detail to include in the diagrams. If too many states or actions were added, the diagrams became difficult to read and understand. On the other hand, if too few were included, important system behaviour was not clearly represented.

To address this, a balance had to be found by including only the most important states and transitions that directly support the system’s functionality. This ensured that the diagrams remained clear while still accurately representing system behaviour.

---

## Aligning Diagrams with Agile User Stories

Another challenge was ensuring that the diagrams aligned with the user stories and sprint tasks created in Assignment 6. Since Agile development focuses on small, manageable tasks, it was important that each workflow and object state could be traced back to a specific user story or requirement.

This required careful review of previous work to make sure that all diagrams were consistent with the system requirements and use cases. The integration step helped ensure that all parts of the system design were connected and aligned.

---

## State Diagrams vs Activity Diagrams

A key learning point from this assignment was understanding the difference between state diagrams and activity diagrams.

State diagrams focus on how a single object changes over time. For example, the Movie object transitions between states such as Available, Reserved, and Rented depending on user actions and system events.

In contrast, activity diagrams focus on workflows and processes. They show how different actions are performed step by step, often involving multiple actors such as users and the system. For example, the Rent Movie workflow shows the entire process from cart selection to payment and confirmation.

Understanding this difference helped in choosing the correct diagram type for each scenario and improved the overall design of the system.

---

## Conclusion

Overall, this assignment improved my understanding of system behaviour modelling and how different diagram types are used in software engineering. It highlighted the importance of clarity, structure, and alignment with requirements when designing systems.

The combination of state transition diagrams and activity diagrams provided a complete view of both object behaviour and system workflows, which is essential for effective system design and future implementation.
