# Reflection – Translating Requirements into Use Cases and Test Cases

During the development of the Aura Reels Movie Rental System, several challenges were encountered when translating stakeholder requirements into use cases and test cases.

One of the main challenges was ensuring that all functional requirements from Assignment 4 were correctly mapped to use cases. While the requirements clearly defined what the system should do, converting them into user interactions required careful thinking about how different actors would actually engage with the system. For example, a simple requirement such as "search for movies" had to be expanded into a complete interaction flow involving input validation, result retrieval, and error handling.

Another challenge was maintaining consistency between the use case diagram, use case specifications, and test cases. Each use case needed to align with both stakeholder needs and system requirements, while also being testable. This required ensuring that each use case had clear preconditions, postconditions, and flows that could be validated through test cases. Any inconsistency between these artifacts could lead to confusion or incomplete system design.

Defining alternative flows was also challenging. It required thinking beyond the normal success scenario and considering possible errors or exceptions, such as invalid login credentials, unavailable movies, or failed payments. These scenarios are important because they ensure the system is robust and can handle real-world situations effectively.

When developing test cases, one of the key challenges was ensuring traceability between requirements and tests. Each test case had to correspond directly to a functional requirement to demonstrate that the system meets its intended functionality. Additionally, writing clear and measurable expected results was important to ensure that system behavior can be evaluated objectively.

Another challenge was designing non-functional test scenarios, particularly for performance and security. These aspects are not always visible to users but are critical for system success. For example, simulating 1000 concurrent users required considering how the system would behave under high load, while security testing required ensuring that sensitive data such as passwords are properly encrypted.

Balancing simplicity and completeness was also important. The system needed to remain manageable for an individual project while still covering all necessary features and quality attributes. This required focusing on core functionality while still addressing key stakeholder concerns such as usability, performance, and security.

Overall, this assignment provided valuable insight into how system requirements are transformed into structured models and testable scenarios. It highlighted the importance of clarity, consistency, and completeness in software engineering documentation. By carefully aligning stakeholders, requirements, use cases, and test cases, the system design becomes more robust, understandable, and easier to implement and validate.
